## Jarvis AutoResearch: Candidate Pool Exhaustion Fix + Expansion

### Diagnosis
The autorun candidate pool is exhausted. The zero-variance/sparse filter in jarvis-runner.js::autoRun() (MIN_NONZERO_COUNT=20) is removing 99.95% of the 257,956 remaining candidates because 705 base indicator keys are flagged as sparse or zero-variance. Almost all remaining candidates are cross-products that involve at least one of these sparse keys, so the pool collapses to 119 items — all of which also fail due to dataset < 50 videos.

### Root cause
The new phrase-family metrics added recently (tension_builder, implicit_promise, progressive_reveal, loop_reinforcer, consequence_language, setup_anchor, outcome_tease, proof_signal, expectation_subversion, knowledge_gap, momentum_language, suspense_bridge, identity_challenge, before_after_frame, fomo_signal, mechanism_reveal, authority_stack, payoff_anticipation, effort_signal, narrative_clock, visual_proof_signal, etc.) have phrase lists that match very few videos (1-17 nonzero out of 370 videos). So every cross-product involving them gets filtered out.

### Fix 2: Lower the sparse filter threshold in jarvis-runner.js
In autoRun(), change MIN_NONZERO_COUNT from 20 to 8. This allows cross-products of moderately rare but real signals through. Also: do NOT filter out candidates where BOTH components are pre-upload phrase metrics from ZYGARNIK_FAMILIES — those should pass through even if each is individually sparse (threshold 3 for pre×pre pairs).

Here is the specific change for the filter block in autoRun():
```javascript
// Keep the skipKeys filter but lower threshold
const MIN_NONZERO_COUNT = 8;  // was 20
const zeroVarKeys = new Set();
const sparseKeys = new Set();
for (const ind of indicators) {
    const ds = ind.dataset || [];
    if (!ds.length) continue;
    const vals = ds.map(d => (d && typeof d === 'object') ? d.value : d).filter(v => v != null);
    if (!vals.length) continue;
    const firstVal = String(vals[0]);
    if (vals.every(v => String(v) === firstVal)) {
        zeroVarKeys.add(ind.key);
    } else {
        const nonzeroCount = vals.filter(v => v !== 0 && v !== 0.0).length;
        if (nonzeroCount < MIN_NONZERO_COUNT) sparseKeys.add(ind.key);
    }
}
const skipKeys = new Set([...zeroVarKeys, ...sparseKeys]);
if (skipKeys.size > 0) {
    const before = pool.length;
    pool = pool.filter(k => {
        const parts = k.split('_x_');
        if (parts.length !== 2) return true; // non-interaction keys always pass
        const [a, b] = parts;
        const aSkip = skipKeys.has(a);
        const bSkip = skipKeys.has(b);
        if (!aSkip && !bSkip) return true; // neither sparse, keep
        // Both sparse: only keep if both are pre-upload (pre x pre pairs have value)
        if (aSkip && bSkip) {
            const layerA = metrics.getCandidateLayer(a);
            const layerB = metrics.getCandidateLayer(b);
            return layerA === 'pre' && layerB === 'pre';
        }
        return false; // one sparse, one not: skip (asymmetric noise)
    });
    log('Zero-variance filter: skipped indicators — ' + zeroVarKeys.size + ' constant-value, ' + sparseKeys.size + ' sparse (<' + MIN_NONZERO_COUNT + ' nonzero); removed ' + (before - pool.length) + ' candidates');
}
```

### Fix 4: Fix biasPool spread overflow (defensive)
In biasPool() in jarvis-metrics.js, replace:
```javascript
if (pi >= pre.length && qi < post.length) { result.push(...post.slice(qi)); break; }
if (qi >= post.length && pi < pre.length) { result.push(...pre.slice(pi)); break; }
```
With safe loops:
```javascript
if (pi >= pre.length && qi < post.length) { for (let ii = qi; ii < post.length; ii++) result.push(post[ii]); break; }
if (qi >= post.length && pi < pre.length) { for (let ii = pi; ii < pre.length; ii++) result.push(pre[ii]); break; }
```
